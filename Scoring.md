### #0 — medmancifra/fastAPI-MCP-server

**Репозиторий:** https://github.com/medmancifra/fastAPI-MCP-server
**Команда:** Бинтанг, Дмитрий
**Тема:** OCR + Barcode Scanner MCP

#### Описание решения

MCP-сервер на FastAPI с двумя инструментами: OCR (Optical Character Recognition) и сканирование штрих-кодов/QR-кодов. Сервер требует JWT-аутентификацию через Descope. Дополнительно реализованы 2 вспомогательных инструмента: анализатор кода и файловая система.

#### Сильные стороны

- Хорошая структура проекта (app/, tools/, auth/)
- Рабочий Docker с поддержкой `serve` и `smoke` команд
- Отличный DEMO.md (212 строк, 5 сценариев, troubleshooting)
- 4 реализованных инструмента (2 основных + 2 бонусных)
- Безопасность: нет секретов, JWT-аутентификация, sandbox для файловой системы
- Тесты (28/42 проходят)
- demo_project/ с тестовыми изображениями

#### Недостатки

| Уровень | Описание |
|---|---|
| 🔴 **Фатальный** | README.md содержит только 7 строк (имена команды). Ни один обязательный раздел не присутствует. Хотя Readme-en.md частично компенсирует это, оценщик видит пустой README. |
| 🟡 **Критический** | Обязательная JWT-аутентификация через внешний сервис Descope нарушает требование «без платных API в базовом сценарии». Без Descope Project ID сервер не запустится. |
| 🟡 **Критический** | Тема (OCR + Barcode) не является «dev-болью» в прямом смысле — это не developer workflow tool. |
| 🟠 **Важный** | Readme-en.md имеет проблемы форматирования (сломанные code blocks) |
| 🟢 **Неважный** | Нет `docker-compose.yml` с ресурсными лимитами по умолчанию |

**Итоговый балл: 48/105**

### #4 — Podtverzhdeno (Agentic-dependency-updater-Hackathon)

**Репозиторий:** https://github.com/Podtverzhdeno/Agentic-dependency-updater-Hackathon
**Тема:** Agentic Dependency Updater (обновление зависимостей через LangGraph/LangChain)

#### Описание решения

Агентная система на LangGraph + LangChain для автоматического обновления зависимостей Python-проектов. 4 агента: Scan Agent, Parse Agent, Process Agent, Report Agent. LLM анализирует риски major-обновлений. Использует SQLite для истории обновлений. Предоставляет MCP-инструменты для каждого агента и оркестровый `run_dependency_update`.

#### Сильные стороны

- Интересная агентная архитектура (LangGraph граф из 4 последовательных агентов)
- MCP-инструменты для каждого агента отдельно + общий оркестратор
- SQLite для персистентной истории (уникальная особенность)
- Генерация Markdown-отчётов об обновлениях
- Присутствует smoke-тест (в entrypoint.py)

#### Недостатки

| Уровень | Описание |
|---|---|
| 🔴 **Фатальный** | **Обязательная зависимость от внешнего платного API (OpenRouter/LLM)**. Без `OPENROUTER_API_KEY` сервер не работает. README явно инструктирует создать API-ключ. Нарушает требование «локальность без платных API». |
| 🔴 **Фатальный** | **Нет HTTP endpoint `/health`**. `health_check` реализован как MCP-инструмент (tool), а не как HTTP endpoint. `docker run IMAGE smoke` не может проверить `/health` без MCP-клиента. |
| 🔴 **Фатальный** | **Нет docker-compose.yml**. Нет ресурсных лимитов (CPU, RAM) в конфигурации. |
| 🟡 **Критический** | README.txt (из Yandex Disk) — полностью искажённая кодировка (UTF-16 вместо UTF-8), не читаемый. README.md в GitHub репозитории тоже минимальный (69 строк), без описания инструментов и параметров. |
| 🟡 **Критический** | `smoke` команда (в entrypoint.py) только тестирует внутренние функции Python, не проверяет реальный HTTP endpoint. Это не полноценный smoke-тест контейнера. |
| 🟡 **Критический** | demo_project/ крайне минимальный — только requirements.txt (3 пакета) + history.db + один старый отчёт. Нет репрезентативного тестового проекта. |

| 🟢 **Неважный** | В репозитории есть файл `identifier.sqlite` и `dependencies.db` — предположительно тестовые артефакты. |

## Примечания к оценке оригинальности

- **#3 Денис (90/100)**: Уникальная комбинация из трёх различных инструментов разных доменов. Особенно оригинален `audit_dockerfile` — редкий инструмент в экосистеме MCP. Воссоздание git-истории прямо в Docker-образе — нестандартное и умное решение.
- **#1 Рене Декарт (85/100)**: Тема инспекции зависимостей и лицензий достаточно популярна (npm audit, pip-audit существуют), но интеграция нескольких экосистем + SBOM-генерация через MCP — оригинально. Стабы для DEMO_MODE — хорошее архитектурное решение.
- **#2 Пифагор (80/100)**: TODO/FIXME сканеры существуют, но интеграция с MCP, ресурсами и промптами (standup_prep) добавляет оригинальности. Специально подготовленный demo_project с реалистичными «плохими» файлами — хорошая деталь.
- **#4 Podtverzhdeno (65/100)**: LangGraph-агенты для обновления зависимостей — интересная идея, но нарушение ключевых требований (локальность, health endpoint) снижает оценку. Агентная архитектура сама по себе оригинальна.
- **#0 medmancifra (55/100)**: OCR + Barcode — функциональные инструменты, но не dev-tools в классическом смысле. Идея понятна, но слабо соответствует теме хакатона «MCP для разработчиков».

## Методология оценки

Оценка проведена по следующим критериям технического чек-листа без развёртывания:

1. **README**: наличие всех обязательных разделов (Что делает, Быстрый старт, Как использовать, Ограничения)
2. **DEMO.md**: воспроизводимый сценарий с 2+ tools
3. **Docker**: поддержка `serve` и `smoke`, порт 8000, `/mcp` и `/health` endpoints
4. **Локальность**: базовый сценарий без платных внешних API
5. **Безопасность**: нет секретов, `.env.example`, `.gitignore`
6. **Инструменты**: минимум 2 dev-инструмента с чёткими интерфейсами
7. **demo_project/**: тестовые данные для воспроизводимой проверки
8. **Ресурсные лимиты**: CPU 2.0, RAM 2048MB в конфигурации

Баллы проставлены исходя из критериев жюри (Продуктовая ценность 25 + Качество MCP 20 + Надёжность 20 + Инженерная дисциплина 20 + Agentic engineering 20 = 105 баллов максимум).
